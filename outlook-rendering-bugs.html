Why Custom Emails Break in Outlook (And How to Fix Them)

It is the quintessential nightmare of every modern marketing team. You spend hours meticulously crafting a gorgeous, highly-converting email design in Figma. Your design team loves it. The client signs off on it. You code it up, send a test to your Gmail and Apple Mail accounts, and it looks flawless.

Then, the marketing director forwards you a screenshot from their corporate desktop PC running Microsoft Outlook 2016.

The margins have collapsed. The beautiful rounded CTA button is now a flat, grey rectangle. Background images have vanished completely into thin air, and a two-column grid has violently stacked into a broken mess. Sound familiar?

If you are a web developer transitioning into email development—or a marketing agency trying to figure out why your templates keep failing—you have undoubtedly hit the Outlook wall. In this article, I am going to explain exactly why custom HTML emails break in Outlook, and more importantly, the technical methodologies professional email developers use to force Outlook to respect our layouts.

The Root of the Problem: Microsoft Word

To understand why Outlook behaves so badly, we have to look back at history. Prior to 2007, Microsoft Outlook used Internet Explorer to render HTML emails. While not perfect, it generally behaved like a standard web browser.

However, starting with Outlook 2007, Microsoft made a drastic change. To unify their Office suite and tighten security, they replaced the Internet Explorer rendering engine with the Microsoft Word rendering engine. Yes, you read that correctly. When you send an HTML email to a modern desktop version of Outlook (2010, 2013, 2016, 2019, and even Office 365 desktop), it is literally using a word processor to draw your code on the screen.

Because Microsoft Word was built for formatting printed documents, not modern web rendering, it lacks support for the vast majority of CSS properties web developers take for granted today.

  • No Flexbox or CSS Grid.
  • Terrible padding and margin support.
  • No native background-image support on block elements.
  • Max-width and Min-width properties are largely ignored.

This is why taking standard website code and pasting it into an email sender will result in an absolute disaster. To fix this, we have to write code specifically tailored to trick Microsoft Word into doing what we want.

The Ultimate Fix: Ghost Tables (Conditional Comments)

Because Outlook ignores properties like `max-width` on standard `<div>` elements, a fluid 600px container will stretch infinitely across an Outlook user's screen. To contain our designs, we have to use HTML tables. But we don't want those rigid tables ruining the responsive experience for mobile users on iOS and Android.

The solution is a technique known as Ghost Tables.

Ghost Tables utilize Outlook-specific conditional comments. These are special HTML comments that every other email client (Gmail, Yahoo, Apple Mail) sees as a standard, invisible comment and ignores. But Outlook reads them as literal, executable code.

Here is an example of what a Ghost Table looks like in raw code:

<!--[if mso | IE]> <table role="presentation" border="0" cellpadding="0" cellspacing="0" width="600" align="center" style="width:600px;"> <tr> <td style="line-height:0px;font-size:0px;mso-line-height-rule:exactly;"> <![endif]--> <div style="margin:0px auto;max-width:600px;"> <!-- Your standard responsive email content goes here --> </div> <!--[if mso | IE]> </td> </tr> </table> <![endif]-->

How this works: Modern clients like Gmail completely ignore the `<!--[if mso]>` tags. They only see the standard `<div>` with a `max-width: 600px`, which allows them to cleanly shrink the design down for mobile phones.

Outlook, however, activates the code inside the brackets. It creates a rigid, 600-pixel wide HTML table, places the content inside of it, and locks the layout in place. By wrapping our columns and sections in these Ghost Tables, we give Outlook the strict, rigid structure it desperately needs without sacrificing mobile responsiveness for the rest of our audience.

Fixing Buttons and Backgrounds with VML

Another major issue with the Microsoft Word rendering engine is how it handles buttons and background imagery. Outlook does not support the CSS `border-radius` property on links, nor does it reliably support CSS background images.

To create a rounded button or place text over a background image in Outlook, we have to utilize VML (Vector Markup Language). VML is a highly outdated XML language created by Microsoft in the late 1990s. It was essentially abandoned by the web development community decades ago, but it still lives deep inside the DNA of Microsoft Office.

If you want a bulletproof button that looks identical in Apple Mail and Outlook, you have to write standard CSS for Apple, and wrap it in a VML `<v:roundrect>` tag specifically for Outlook.

<!--[if mso]> <v:roundrect xmlns:v="urn:schemas-microsoft-com:vml" xmlns:w="urn:schemas-microsoft-com:office:word" href="https://yourlink.com" style="height:40px;v-text-anchor:middle;width:200px;" arcsize="10%" stroke="f" fillcolor="#0047AB"> <w:anchorlock/> <center style="color:#ffffff;font-family:sans-serif;font-size:16px;font-weight:bold;"> Click Here </center> </v:roundrect> <![endif]-->

This VML block tells the Microsoft Word engine to literally draw a vector rectangle with rounded corners, fill it with a blue color, and lock the text anchor to the center. It is a highly manual, tedious process, but it is the absolute only way to guarantee design parity across B2B desktop environments.

CSS Quirks You Must Remember

Beyond Ghost Tables and VML, there are several CSS quirks you must memorize when developing for Outlook:

  • No CSS Shorthand: Outlook struggles with shorthand. Never write `margin: 10px 0 20px 0;`. You must explicitly declare `margin-top: 10px; margin-bottom: 20px;`.
  • The 1px Line Height Bug: If you have an empty table cell used for spacing, Outlook will force it to be at least the height of the default text font (usually 15px). To create a 5px spacer, you must apply `line-height: 5px; font-size: 5px; mso-line-height-rule: exactly;` to the table cell.
  • Web Fonts Fallbacks: Outlook famously chokes on Google Fonts. If you import a web font, Outlook will frequently ignore your fallback font (like Arial) and revert entirely to Times New Roman. You must use specific `mso-font-alt` CSS declarations to prevent this.

The MJML Framework Solution

Writing raw HTML tables and VML by hand for an entire newsletter takes hours of highly specialized labor. This is why modern professional email developers rely heavily on MJML (Mailjet Markup Language).

MJML is a semantic syntax that you compile into HTML. You write clean, tag-based code (like `<mj-button>`), and the MJML engine automatically generates the hundreds of lines of Ghost Tables and VML code required to make it work in Outlook.

However, an email developer must still understand the underlying raw HTML. When a layout breaks, or a custom ESP requires dynamic logic injected into the middle of a Ghost Table, you have to be able to open the raw compiled HTML and manually adjust the conditional comments without breaking the table structure.

Conclusion: Stop Guessing, Start Testing

Building custom emails is an entirely different discipline than standard web development. It requires an intimate understanding of legacy systems, rendering engines, and rigorous quality assurance testing.

You cannot rely on sending a test email to your personal Gmail account and assuming it looks good everywhere. Professional developers run their code through platforms like Litmus or Email on Acid to generate live screenshots across 50+ real devices and email clients before the code is ever handed over to a client.
